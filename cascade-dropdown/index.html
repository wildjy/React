Cascade Dropdown 학습 샘플

대학 → 계열 → 학과 순서로 상위 선택이 하위 옵션을 결정하는 연쇄 의존 드롭다운 패턴입니다. 왼쪽에서 직접 선택해 보고, 오른쪽에서 내부 상태와 캐시가 어떻게 변하는지 실시간으로 확인하세요.

POINT 01

IoC (제어의 역전)

함수를 prop / return 값으로 주입해서 hook 의 동작을 외부에서 결정. fetcher 만 갈아끼우면 대학·고교·지역 등 어디든 같은 hook 재사용.

const onUniversityChange = async (id) => {...}
POINT 02

Cascade (연쇄 의존)

상위 선택값이 하위 옵션을 결정. 상위가 바뀌면 하위 선택값·옵션을 반드시 리셋해야 잘못된 조합이 남지 않음.

setSelected({ ..., majorTypeCode: "", majorId: "" })
POINT 03

상태 vs 캐시

화면에 그릴 옵션은 useState (리렌더 필요), 내부 계산용 raw 데이터는 useRef (리렌더 불필요). 역할이 다름.

useRef<Majors[]>([]) // 화면에 직접 안 그림

      A

      타입 정의 — 데이터 모양 계약

      패턴의 첫 단계는 "데이터가 어떤 모양으로 흐를 것인가" 를 먼저 합의하는 것. TypeScript 타입이 곧 그 합의서입니다.

      1// 모든 드롭다운이 공통으로 쓰는 옵션 모양
      2interface Option {
      3  label: string;
      4  value: string;
      5}
      6 
      7// 서버가 대학 ID 로 돌려주는 응답 모양
      8interface University {
      9  universityId: string;
      10  universityName: string;
      11  majors: {
      12    majorId: string;
      13    majorName: string;
      14    majorTypeCode: string;
      15  }[];
      16}
      17 
      18// IoC 의 핵심 — fetcher 의 함수 시그니처 계약
      19// 대학 ID 받아 → 계열 옵션 list 를 비동기 반환
      20type FetchMajorTypesByUniv =
      21  (universityId: string) => Promise<Option[]>;
      Option
      모든 드롭다운이 공통으로 쓰는 { value, label } 형태. <select> 와 1:1 대응.
      University
      가짜 서버 응답 모양 — 대학 1개 + 그 대학이 운영하는 학과 list.
      FetchMajorTypesByUniv
      IoC 의 핵심 — fetcher 함수의 시그니처 계약. 이걸 prop 으로 주입하면 hook 재사용 가능.
      B

      Mock 데이터 + 가짜 서버

      실무에선 backend DB 가 가진 정보입니다. 학습용으로 메모리에 미리 넣어두고 setTimeout 으로 네트워크 지연만 흉내냈습니다.

      MOCK_UNIVERSITIES— 가짜 서버 DB (실제론 backend 가 가진 정보)
      id=1서울대학과 4개
      └H 인문국문학과m1
      └H 인문사학과m2
      └N 자연물리학과m3
      └N 자연컴퓨터공학과m4
      id=2연세대학과 3개
      └H 인문영문학과m5
      └A 예체능체육학과m6
      └Z 자율자유전공학부m7
      id=3고려대학과 3개
      └H 인문경제학과m8
      └N 자연생명과학과m9
      └A 예체능디자인학과m10
      ALL_MAJOR_TYPES— 전역 코드 테이블 (모든 대학 공통)
      H 인문N 자연A 예체능Z 자율

      그리고 그 DB 를 조회하는 가짜 서버 함수 — 실무에선 이 자리에 ky/axios 호출이 들어갑니다:

      1// 실제로는 ky/axios 로 서버 호출
      2// 여기선 setTimeout 으로 네트워크 지연 흉내만 냄
      3async function mockFetchUnivMajors(universityId: string) {
      4  await new Promise(r => setTimeout(r, 300));  // 300ms 지연
      5  const found = MOCK_UNIVERSITIES.find(
      6    u => u.universityId === universityId,
      7  );
      8  if (!found) throw new Error(`Univ ${universityId} not found`);
      9  return found.majors;
      10}
      C

      Cascade Hook — 전체 구조

      이 hook 이 패턴의 핵심. 컴포넌트는 얇게 두고 모든 로직을 여기로 모읍니다. 핸들러 본문은 D 섹션에서 따로 봅니다.

      1function useCascadeDropdowns() {
      2  // ─── 상태 (useState) — 바뀌면 화면 다시 그려야 함 ───
      3  const [majorTypeOptions, setMajorTypeOptions] = useState<Option[]>([]);
      4  const [majorOptions,     setMajorOptions]     = useState<Option[]>([]);
      5  const [selected,         setSelected]         = useState({
      6    universityId: '', majorTypeCode: '', majorId: '',
      7  });
      8 
      9  // ─── 캐시 (useRef) — 바뀌어도 화면 안 그려도 됨 ───
      10  const cachedMajorsRef = useRef<University['majors']>([]);
      11 
      12  // ─── 핸들러 3종 (D 섹션에서 본문 자세히) ───
      13  const onUniversityChange = async (id: string) => { /* D-1 */ };
      14  const onMajorTypeChange  = (code: string)     => { /* D-2 */ };
      15  const onMajorChange      = (id: string)       => { /* D-3 */ };
      16 
      17  return {
      18    selected,
      19    universityOptions, majorTypeOptions, majorOptions,
      20    onUniversityChange, onMajorTypeChange, onMajorChange,
      21  };
      22}
      관찰 포인트: useState 3개 (옵션 2개 + 선택값 1개) + useRef 1개 (캐시). useRef 는 화면에 직접 그리지 않는 데이터로, 다음 계열 변경 때 filter 입력으로만 쓰입니다.
      D

      이벤트 핸들러 3종 — 실제 트리거 함수

      각 핸들러는 <DropDown onChange> 에 연결됩니다. 위 데모를 조작하면 정확히 이 함수가 실행되고, 동작 로그(③)에 한 줄씩 찍힙니다.

      D-1

      대학 변경 → onUniversityChange (가장 무거운 핸들러)

      1const onUniversityChange = async (universityId: string) => {
      2  // 1. 선택값 업데이트 + 하위 cascade 리셋 (이전 선택 무효)
      3  setSelected({ universityId, majorTypeCode: '', majorId: '' });
      4  setMajorOptions([]);
      5  setMajorTypeOptions([]);
      6 
      7  if (!universityId) {
      8    cachedMajorsRef.current = [];
      9    return;
      10  }
      11 
      12  // 2. 서버에서 그 대학의 학과 list fetch
      13  const majors = await mockFetchUnivMajors(universityId);
      14  cachedMajorsRef.current = majors;  // ← 캐시 저장 (리렌더 X)
      15 
      16  // 3. 가져온 학과들의 majorTypeCode set 으로
      17  //    → 그 대학에 존재하는 계열만 옵션화
      18  const availableTypeCodes = new Set(majors.map(m => m.majorTypeCode));
      19  const filteredTypes = ALL_MAJOR_TYPES.filter(t =>
      20    availableTypeCodes.has(t.value),
      21  );
      22  setMajorTypeOptions(filteredTypes);
      23};
      왜 cascade 리셋이 fetch 보다 먼저인가? 사용자가 fetch 응답 대기 중에 다른 대학을 누를 수 있기 때문. 이전 계열/학과 옵션을 즉시 비워서 잘못된 조합이 화면에 잠시도 안 남게 합니다.
      D-2

      계열 변경 → onMajorTypeChange (서버 호출 없음)

      1const onMajorTypeChange = (majorTypeCode: string) => {
      2  setSelected(prev => ({ ...prev, majorTypeCode, majorId: '' }));
      3 
      4  if (!majorTypeCode) {
      5    setMajorOptions([]);
      6    return;
      7  }
      8 
      9  // 서버 호출 없이 cachedMajorsRef 만 filter ← 캐시의 핵심 가치
      10  const filteredMajors = cachedMajorsRef.current
      11    .filter(m => m.majorTypeCode === majorTypeCode)
      12    .map(m => ({ value: m.majorId, label: m.majorName }));
      13 
      14  setMajorOptions(filteredMajors);
      15};
      캐시의 핵심 가치: 같은 대학 안에서 계열을 이리저리 바꿔도 서버를 다시 두드리지 않습니다. cachedMajorsRef 에 학과 raw 데이터가 있기 때문에 client 에서 즉시 filter.
      D-3

      학과 변경 → onMajorChange (cascade 끝)

      1const onMajorChange = (majorId: string) => {
      2  // 하위 cascade 가 없으니 선택값만 저장
      3  setSelected(prev => ({ ...prev, majorId }));
      4};
      가장 단순한 핸들러: 더 이상 하위 cascade 가 없으니 선택값만 저장. 폼 제출은 보통 별도 버튼에서 selected 객체를 통째로 전송합니다.

      ⑤ 동작 흐름 — 단계별로 무엇이 일어나는가

      왼쪽 데모를 조작할 때마다 이 순서대로 함수가 실행됩니다. 동작 로그와 비교해 보세요.

      1

      대학 변경 → onUniversityChange

      1. 1.setSelected({ universityId, majorTypeCode: "", majorId: "" }) — 하위 cascade 리셋
      2. 2.setMajorOptions([]) / setMajorTypeOptions([]) — 옵션 비움
      3. 3.await mockFetchUnivMajors(id) — 가짜 서버 호출 (300ms 지연)
      4. 4.cachedMajorsRef.current = majors — 응답을 캐시에 저장 (리렌더 X)
      5. 5.majors → majorTypeCode set 추출 → ALL_MAJOR_TYPES 와 교집합 → setMajorTypeOptions
      2

      계열 변경 → onMajorTypeChange

      1. 1.setSelected(prev → { ...prev, majorTypeCode, majorId: "" }) — 학과만 리셋
      2. 2.서버 호출 없음 — cachedMajorsRef.current 를 majorTypeCode 로 filter
      3. 3.filter 결과를 { value, label } 로 변환 → setMajorOptions
      3

      학과 변경 → onMajorChange

      1. 1.하위 cascade 없음 — setSelected(prev → { ...prev, majorId }) 만 실행
      2. 2.이후 폼 제출·서버 전송 등에서 selected 객체를 통째로 사용

      ⑥ 왜 useState 와 useRef 를 나눠 썼나?

      같은 "데이터 보관"이라도 화면에 영향을 주는지에 따라 도구가 달라집니다.

      useState · majorTypeOptions / majorOptions

      값이 바뀌면 화면을 다시 그려야 함.<select>안의 옵션 목록이 그것.

      → setState 가 트리거하는 리렌더가 필수.

      useRef · cachedMajorsRef

      값이 바뀌어도 화면이 바뀌지 않음. 다음 계열 변경 때 filter 입력으로만 쓰는 raw 데이터.

      → useState 로 두면 불필요한 리렌더 1회 추가. useRef 가 더 정확.

      판단 기준: "이 값이 바뀌었을 때 화면 어딘가가 달라져야 하나?" → Yes 면 useState, No 면 useRef. 캐시·이전값 기록·외부 객체 참조 등은 보통 useRef.

      ⑦ 다음 단계 — 실무에서 IoC 로 확장

      지금은 hook 안에 mockFetchUnivMajors 가 박혀 있습니다. 실무 코드는 fetcher 를 prop 으로 받아서 hook 을 여러 도메인(대학·고교·지역)에 재사용할 수 있게 분리합니다.

      // Before — fetcher 가 hook 안에 박혀 있음 (학습용)
function useCascadeDropdowns() {
  const majors = await mockFetchUnivMajors(id);
  ...
}

// After — fetcher 를 prop 으로 주입 (실무용)
function useCascadeDropdowns(fetcher: FetchMajorTypesByUniv) {
  const majors = await fetcher(id);   // 호출부만 바꿔치기 가능
  ...
}

// 사용처: 대학용·고교용·지역용 fetcher 만 갈아끼우면 같은 hook 재사용
